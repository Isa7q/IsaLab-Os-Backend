Guia de Integração e APIs

IsaLab Homelab OS — API REST Nativa e Assíncrona em Rust

🚀 Standalone & Unificado

O backend em Rust foi desenvolvido para atuar de forma unificada: ele expõe a API REST no endpoint prefixado por /api e, simultaneamente, serve os arquivos estáticos de produção do frontend na mesma porta (padrão: 3000). Isso elimina qualquer necessidade de configurações complexas de CORS ou proxies de desenvolvimento!

1. Visão Geral e Inicialização

A API consome e altera o banco de dados SQLite persistente local homelab.db. Toda a autenticação é gerida via token JWT assinado em tempo de execução e as senhas do operador são tratadas com criptografia unidirecional Bcrypt.

Método Rota Tipo Função
GET /api/status Público Verifica se o sistema já passou pelo setup/onboarding inicial.
POST /api/onboard Público Salva os dados do onboarding inicial e gera a senha master.
POST /api/auth/login Público Autentica o administrador e gera o token JWT.
GET /api/services Público Lista todos os serviços cadastrados no diretório SQLite.
POST /api/pipelines/run Protegido Dispara o fluxo de implantação assíncrono em background.

2. Detalhamento de Endpoints

GET /api/status
Público

Usado na inicialização do frontend. Retorna se o Wizard de primeiro acesso precisa ser exibido e as configurações atuais.

Resposta (200 OK)
{
  "onboarded": false,
  "localIp": "",
  "piholePath": "",
  "hasNpmToken": false
}
POST /api/onboard
Público

Enviado pelo formulário do Wizard de primeiro acesso. Registra os caminhos e cria a senha mestra.

Corpo da Requisição (JSON)
{
  "localIp": "192.168.1.50",
  "piholePath": "/app/big-bear-pihole/etc/pihole.toml",
  "npmToken": "jwt-token-npm",
  "adminPassword": "minha_senha_mestra"
}
Resposta Sucesso (200 OK)
{
  "success": true,
  "message": "Onboarding realizado com sucesso!"
}
POST /api/auth/login
Público

Valida as credenciais administrativas do painel contra o banco.

Corpo da Requisição (JSON)
{
  "password": "senha_digitada"
}
Resposta Sucesso (200 OK)
{
  "success": true,
  "token": "isa-homelab-admin-auth-token-jwt-string"
}
Resposta Erro (401 Unauthorized)
{
  "error": "Senha de administrador inválida."
}
POST /api/pipelines/run
Protegido

Dispara o pipeline automatizado em background utilizando Tokio Tasks. Retorna imediatamente um ID de controle para o frontend fazer consultas periódicas de logs.

Corpo da Requisição (JSON)
{
  "serviceName": "Komga Comics",
  "subdomain": "komga.isa7q.uk",
  "ip": "192.168.1.50",
  "port": 8080,
  "description": "Servidor de quadrinhos",
  "category": "Media",
  "registerNPM": true,
  "registerPihole": true,
  "createDocker": true
}
Resposta (200 OK)
{
  "success": true,
  "pipelineId": "pipe-1718361000000"
}

3. O Motor de Pipelines Assíncronos

Ao receber a requisição de execução de pipeline, o backend Rust executa as seguintes etapas não-bloqueantes:

  1. Persistência de Metadados: Estrutura o log inicial e o status running no SQLite.
  2. Modificação do Pi-hole TOML: Lê de forma segura o pihole.toml, localiza o array [dns].hosts, remove duplicatas, insere a nova linha "IP SUBDOMÍNIO" e executa o comando assíncrono pihole restartdns.
  3. Pipeline NPM: Autentica na API do Nginx Proxy Manager no IP configurado, gera o token JWT temporário e envia o payload para cadastrar a regra de proxy e SSL.
  4. Docker Container: Cria o contêiner virtualizado no SQLite com labels personalizadas e status running.
  5. API do CasaOS: Intercepta o ID do serviço e consulta a API oficial do CasaOS na porta 88 para extrair a URL direta de seu ícone, atualizando a coluna icon_url.
💡 Dica para Integração do Frontend

Como a execução da pipeline roda em uma thread paralela do Tokio, o frontend deve fazer polling do endpoint GET /api/pipelines a cada 1.5s para capturar os logs incrementais e atualizar a janela de progresso na tela do operador.
