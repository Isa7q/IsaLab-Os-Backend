Introdução & Estratégia

A implantação do ecossistema do IsaLab OS em ambiente de produção no Homelab segue a regra clássica de segurança: ocultamento de portas sob redes virtuais privadas.

Ao invés de expor portas aleatórias do host como 3001 (Rust) ou 5173 (Vite/Node) para a sua rede local (10.0.0.x), implementou-se um túnel de comunicação interna através de uma Docker Network privada. Apenas o Nginx Proxy Manager (NPM) tem suas portas físicas de borda expostas para a rede local, atuando como o único ponto de entrada para o ecossistema.

Arquitetura de Portas & Redes

Os contêineres do frontend e do backend operam na rede isolada proxy-network. Os nomes dos serviços no Docker Compose são resolvidos pelo DNS interno do Docker, permitindo que os containers se conversem de forma transparente.

Arquivos de Configuração Criados

Abaixo estão os arquivos exatos configurados e utilizados para orquestrar o deploy de produção do ecossistema.

1. Docker Compose Principal

Localizado na raiz do projeto (/app/isalab-os/docker-compose.yml), este arquivo gerencia o ciclo de vida dos containers.

docker-compose.yml 
version: '3.8'

services:
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    image: homelab-frontend:latest
    container_name: homelab-frontend
    networks:
      - proxy-network
    restart: always

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    image: homelab-backend-rust:latest
    container_name: homelab-backend
    environment:
      - PORT=3001
      - DATABASE_URL=sqlite:///app/data/homelab.db
      - JWT_SECRET=isa-secret-token-key-123 # Altere para uma chave segura no deploy real
    volumes:
      # Persistência do Banco de Dados SQLite
      - ./backend/data:/app/data
      # Bind físico com as configurações do Pi-hole v6
      - /app/big-bear-pihole/etc/pihole.toml:/etc/pihole/pihole.toml
      # Acesso ao Docker do Host para monitorar e gerenciar outros containers em tempo real
      - /var/run/docker.sock:/var/run/docker.sock
    networks:
      - proxy-network
    restart: always

networks:
  proxy-network:
    external: true # Usa a rede externa onde o NPM já está conectado

2. Configuração Nginx do Frontend (SPA + Reverse Proxy)

Localizado em /app/isalab-os/frontend/nginx.conf. Esse arquivo atua como o roteador SPA e faz o proxy reverso interno das requisições /api/* para o backend.

nginx.conf 
server {
    listen 80;
    server_name localhost;

    location / {
        root /usr/share/nginx/html;
        index index.html index.htm;
        try_files $uri $uri/ /index.html;
    }

    # Proxy para o backend Rust
    location /api/ {
        proxy_pass http://homelab-backend:3001/api/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;

        # Configurações de timeout para operações longas (ex: pipelines)
        proxy_connect_timeout 300s;
        proxy_send_timeout 300s;
        proxy_read_timeout 300s;
    }
}

3. Dockerfile Multiestágio do Backend Rust

Localizado em /app/isalab-os/backend/Dockerfile. Compila o executável da API no Rust 1.96 e gera uma imagem final otimizada baseada no Debian Trixie.

Dockerfile 
# Stage 1: Build
FROM rust:1.96-slim AS builder
WORKDIR /app
RUN apt-get update && apt-get install -y pkg-config libssl-dev && rm -rf /var/lib/apt/lists/*
COPY Cargo.toml Cargo.lock ./
COPY migrations ./migrations
COPY src ./src
RUN cargo build --release

# Stage 2: Runtime
FROM debian:trixie-slim AS runtime
WORKDIR /app
RUN apt-get update && apt-get install -y ca-certificates libssl3 sqlite3 && rm -rf /var/lib/apt/lists/*
# Copiar a CLI do Docker para permitir gerenciamento de containers a partir do socket
COPY --from=docker:latest /usr/local/bin/docker /usr/local/bin/docker
COPY --from=builder /app/target/release/homelab-governance-api /app/homelab-governance-api
COPY migrations /app/migrations
ENV PORT=3001
ENV DATABASE_URL=sqlite:///app/data/homelab.db
EXPOSE 3001
CMD ["/app/homelab-governance-api"]

Resolução de Problemas (Troubleshooting)

Durante a migração para a infraestrutura de produção em Docker, identificamos e contornamos dois obstáculos significativos:

1. Incompatibilidade com GLIBC 2.39 no Debian

O compilador do Rust 1.96 gerou um binário dinamicamente linkado que requeria a biblioteca GLIBC 2.39 (induzido por crates que exigem a Edition 2024 como a dependência transitiva time). A imagem base de execução original, debian:bookworm-slim, contava apenas com a versão GLIBC 2.36, gerando erros de execução imediatamente.

Solução: Atualizamos o estágio final de runtime no Dockerfile para utilizar debian:trixie-slim (a versão testing do Debian), que fornece nativamente suporte a GLIBC 2.39+.

2. Caminhos de Bancos de Dados SQLite

A API tentava tratar o caminho da URL de banco SQLite removendo barras iniciais (trim_start_matches('/')). Com isso, a URL de banco absoluto no container (sqlite:///app/data/homelab.db) era convertida em app/data/homelab.db, fazendo com que o Rust tentasse criar e abrir o arquivo sob o subdiretório relativo /app/app/data/..., o que resultava no erro (code: 14) unable to open database file.

Solução: Refatoramos a lógica de inicialização de main.rs para verificar se a URL é absoluta (prefixo sqlite:///) e manter a barra inicial no caminho resultante do arquivo.

3. Monitoramento em Tempo Real do Host

Como a API roda dentro de um container isolado, os comandos docker ps falhavam por padrão. Nós solucionamos isso instalando a CLI do Docker no estágio de runtime e efetuando a montagem física do soquete /var/run/docker.sock:/var/run/docker.sock no Docker Compose. Com isso, a listagem dos contêineres reais do Homelab voltou a operar em tempo real.

4. Telemetria Real de Hardware (Sem Dados Mockados)

Substituímos toda a simulação estática de métricas por telemetria ativa. O backend lê diretamente os arquivos virtuais do kernel do host montados no container (/proc/stat para CPU, /proc/meminfo para RAM, /proc/uptime para Uptime e /proc/net/dev para tráfego de rede) e executa o utilitário df para medir o espaço de disco real do host, fornecendo dados dinâmicos do Homelab.

5. Automações de Implantação Dinâmicas (Pipelines Reais)

Substituímos credenciais estáticas de e-mail e mocks de criação do Docker. O pipeline lê dinamicamente as credenciais de e-mail e senha do NPM configuradas no onboarding e dispara comandos reais de criação de contêineres (docker run) via socket, associando os novos contêineres diretamente à rede privada do Homelab.

Comandos de Operação Úteis

Aqui estão os comandos necessários para interagir com a infraestrutura no dia a dia:

Inicializar e Reconstruir

Terminal
# Subir os containers em background forçando a reconstrução
docker compose up -d --build

# Parar os containers sem apagar os volumes
docker compose down

Monitorar Logs do Backend

Terminal
# Acompanhar logs da API do backend em tempo real
docker logs -f homelab-backend

Criar Rede Manual e Conectar o NPM (Passo inicial feito)

Terminal
# Criar a rede compartilhada
docker network create proxy-network

# Conectar o container do Nginx Proxy Manager à rede criada
docker network connect proxy-network nginxproxymanager
